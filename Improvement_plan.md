# 改善計画メモ

## 概要
- 現状の実装は `pytest` と `ruff check` を通過している。
- ただし、実データを用いた確認では、解答 Markdown の形式ゆれにより正解番号を抽出できない問題が複数存在した。
- その結果、`correct_number == 0` の問題が生成され、ユーザーが正答しても不正解扱いになる不具合がある。

## 確認した問題点

### 1. 正解番号の抽出漏れ
- 対象: `src/quiz_app/parser.py`
- 原因:
  - `正解: 4`
  - `**4**`
  には対応しているが、
  - `正解の選択肢は **4** です`
  のような実データ形式を拾えていない。
- 影響:
  - 第10回: 11問
  - 第4回: 13問
  - 第6回: 9問
  - 第8回: 11問
  - 合計 44問で `correct_number == 0`

### 2. 解析失敗の握りつぶし
- 対象: `src/quiz_app/parser.py`
- 原因:
  - `merge_answers()` が答え未取得時に `(0, "")` を設定する。
  - `load_quiz_data()` がその不整合を例外化していない。
- 影響:
  - 壊れた問題がそのままアプリに渡る。

### 3. UI での例外未処理
- 対象: `src/quiz_app/app.py`
- 原因:
  - `load_quiz_data()` の `QuizParseError` を捕捉していない。
- 影響:
  - データ不備時にユーザー向けのエラーメッセージではなく、アプリ全体のエラーになりうる。

### 4. import 時の副作用
- 対象: `src/quiz_app/app.py`
- 原因:
  - `main()` がモジュール末尾で無条件実行されている。
- 影響:
  - 補助スクリプトや検証コードから import しただけで Streamlit 実行が始まる。

## 改善方針

### 方針A（採用）
- `parser.py`
  - 正解番号抽出のパターンを実データの形式ゆれに対応させる。
  - 正解未抽出や、抽出した正解番号が選択肢に存在しない場合は `QuizParseError` を送出する。
- `app.py`
  - `QuizParseError` を捕捉し、`st.error()` で画面表示する。
  - `main()` は `if __name__ == "__main__":` でのみ実行する。
- `tests/`
  - 実データ形式の回帰テストを追加する。
  - 不整合データの例外テストを追加する。
  - UI のエラー表示テストを追加する。

### 方針B（今回は不採用）
- 解答 Markdown を行単位の状態遷移パーサで全面的に再設計する。
- 将来的には有効だが、今回の修正対象としては差分が大きい。

## 実装タスク
1. parser の回帰テストを追加し、実データ形式の抽出失敗を再現する。
2. parser 実装を最小修正で更新し、正解抽出と不整合検知を行う。
3. app の UI エラー表示テストを追加する。
4. app 実装を更新し、解析エラー時に画面表示で復帰させる。
5. `main()` の起動条件を見直し、import 副作用を減らす。
6. `uv run ruff check src/ tests/` と `uv run pytest` で検証する。
